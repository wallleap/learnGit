Git 学习

1、初始化环境

 下载 Git、注册 GitHub 等账号

        # 查看 git 版本
        git --version

 配置本地 Git 信息

  • git config --global user.name "Your Name"
  • git config --global user.email "email@example.com"
  •             # 查看配置信息（按 q 退出）
            git config --list

添加了 --global 是针对全局生效的

如果某个仓库需要单独的，只需在这个仓库中使用没有 --global 的命令配置一遍即可，或把 --global 换成 --local

 生成 keygen

ssh-keygen -t rsa -C "email@example.com"

或使用 ed25519 算法

ssh-keygen -t ed25519 -C "SSH Key"

将生成的 ~/.ssh/id_rsa.pub / ~/.ssh/id_ed25519.pub 中的公钥放到需要提交的服务器上（追加到 ~/.ssh/authorized_keys 文件中）或者在线平台的设置 SSH keys 新建一个，拷贝公钥内容上去

 测试连接（例如 GitHub）

ssh -T git@github.com

输出如下内容就代表成功

Hi username! You've successfully authenticated, but GitHub does not provide shell access.

 同一平台，本地使用多个账号

生成多个密钥对（同一平台不能出现相同的公钥）

        ssh-keygen -t ed25519 -C "work@example.com" -f ~/.ssh/id_ed25519_work
        ssh-keygen -t ed25519 -C "personal@example.com" -f ~/.ssh/id_ed25519_personal

将生成的 ~/.ssh/id_ed25519_work.pub 和 ~/.ssh/id_ed25519_personal.pub 中的公钥内容放到需要提交的服务器上（追加到 ~/.ssh/authorized_keys 文件中）或者在线平台的设置 SSH keys 新建一个，拷贝对应账号的公钥内容上去

配置 SSH 规则

        # 编辑 ~/.ssh/config 文件
        vi ~/.ssh/config

添加如下内容

        # 工作账号
        Host work-github # 最好保留一个 github.com 以便匹配到这个主机名
          HostName github.com
          IdentityFile ~/.ssh/id_ed25519_work
          User git
          IdentitiesOnly yes
        
        # 个人账号
        Host personal-github
          HostName github.com
          IdentityFile ~/.ssh/id_ed25519_personal
          User git
          IdentitiesOnly yes

例如，原链接 git@github.com:username/repo.git 使用工作账号 clone 仓库

git clone git@work-github:username/repo.git

或使用个人账号

git clone git@personal-github:username/repo.git

就是把主机名替换掉，根据配置连接时会使用不同的私钥

测试也要使用对应的主机名

ssh -T git@work-github

如果是另一个平台，那可以接着使用相同的密钥对，例如 id_ed25519_work 在 GitHub 上上传了对应的公钥，在 GitLab 上也可以上传这个的公钥

配置 GitLab 规则 ~/.ssh/config

        # 工作账号
        Host gitlab.com
          HostName gitlab.com
          IdentityFile ~/.ssh/id_ed25519_work
          User git
          IdentitiesOnly yes

使用 git@gitlab.com:username/repo.git 克隆仓库时，会使用 id_ed25519_work 这个密钥对

2、了解 Git 基本命令

工作区（working tree），暂存区（index /stage），本地仓库（repository），远程仓库（remote）

 本地仓库

初始化本地仓库

git init

创建 .gitignore 文件

        touch .gitignore
        # 修改这个文件，将需要忽略的文件名/目录写入
        vi .gitignore

查看仓库中文件状态

        git status

 暂存区

添加当前目录下文件到暂存区

        git add .
        # 添加部分文件
        git add 文件名|文件夹
        # 从暂存区删除文件，但保留工作区文件
        git rm --cached 文件名|文件夹
        # 恢复指定文件 restore 或用 checkout
        git restore 文件名|文件夹
        git restore .

将暂存区中的文件进行提交

        # 直接在 -m 后写提交说明
        git commit -m "本次提交的说明"
        # 使用默认编辑器编写提交说明，保存后关闭文件即可
        git commit
        # 修正最近一次提交（会保留原提交内容，仅补充文件/修改提交信息）
        git commit --amend

 不提交 保留修改（到存储库）

        git stash -a
        # 查看 stash 列表
        git stash list
        # 恢复并删除 stash
        git stash pop stash编号

 远程仓库

添加远程仓库

        git remote add origin(名字) 仓库链接(ssh或https)

查看远程仓库别名

        git remote
        # 查看远程仓库链接
        git remote -v

修改远程仓库别名

        git remote rename origin(原名) new-origin(新名)

删除远程仓库别名

        git remote remove origin(名字)

推送本地 main 分支代码到远程 main 分支

        git push origin main:main
        # origin 是远程仓库别名、前一个 main 是本地分支名、后一个 main 是远程分支名，相同时可省略后半部分
        # 第一次可以多加一个参数 -u，将设置默认推送目标，后续可以直接使用 git push
        git push -u origin main
        # 还有其他参数，-f 可以强制推送，--all 推送所有分支，--tag 推送所有标签

拉取远程代码到工作区

        git pull origin main:main
        # origin 是远程仓库别名、前一个 main 是远程分支名、后一个 main 是本地分支名
        # git pull = git fetch + git merge
        # --rebase 参数使用变基，-r 禁用变基强制使用合并
        # fetch 下载最新更改到本地仓库，但不自动合并到当前分支
        git fetch origin main

克隆远程代码到本地

        git clone 仓库链接 目录名
        # -b main 指定克隆分支，--depth 1 指定克隆深度

 查看文件修改内容

        git diff 文件名

 查看历史记录

        git log
        git log --pretty=oneline

 查看引用日志

        git reflog

 版本回退

        # 回退到上一个版本(Git 中 HEAD 代表当前版本)
        git reset --hard HEAD^
        # 回退到上上个版本
        git reset --hard HEAD^^
        # 回退到上100个版本
        git reset --hard HEAD~100
        # 根据版本号回退
        git reset --hard 版本号前几位或写全
        # 撤销指定版本
        git revert 版本号前几位或写全

如果需要“回到”某个版本，应该使用如下操作（从指定版本恢复文件到工作区）

        git restore --source 版本号前几位或写全 .

 标签

        # 列出所有 tag
        git tag
        # 在当前版本新建 tag
        git tag 标签名
        # 带注释的标签
        git tag -a 标签名 -m "注释"
        # 在某个版本新建 tag
        git tag 标签名 版本号
        # 推送 tag 到远程仓库
        git push origin 标签名
        # 推送所有 tag
        git push origin --tags
        # 删除 tag
        git tag -d 标签名
        # 删除远程 tag
        git push origin --delete 标签名
        git push origin :标签名 # 简写形式

 分支管理

        # 创建分支
        git branch 分支名
        # 切换分支
        git checkout 分支名
        git switch 分支名
        # 可以简写为一条
        git checkout -b 分支名
        git switch -c 分支名
        # 切换到上一分支
        git switch -
        # 查看分支，-r 参数查看远程分支，-a 参数查看所有分支，-v 参数查看分支+最后一次提交信息
        git branch
        # 合并分支到当前分支
        git merge 分支名
        # 挑选某个提交合并到当前分支
        git cherry-pick 版本号
        # 推送分支到远程仓库
        git push origin 分支名
        # 删除分支，-d 参数删除已合并分支，-D 参数强制删除本地分支
        git branch -d 分支名
        # 删除远程分支
        git push origin --delete 分支名
        git push origin :分支名 # 简写形式

3. 冲突解决

当在不同分支上对同一个文件同一行（或相邻行）代码进行了不同的修改，合并/变基时（git merge / git rebase）就会发生冲突（CONFLICT），这时候需要手动解决冲突。 

        # 查看冲突文件
        git status
        # Unmerged paths:
        #   (use "git add/rm ..." as appropriate to mark resolution)
        #   both modified:  文件名

打开冲突文件，会看到类似如下内容

        # 冲突部分
        <<<<<<< HEAD
        当前分支的代码
        =======
        待合并分支的代码
        >>>>>>> 分支名

解决冲突的方法有很多种，可以将冲突部分的内容修改为自己需要的内容，然后删除 <<<<<<< HEAD、 =======、 >>>>>>> 分支名 这三行，最后提交。

或者使用编辑器（如 VS Code）打开文件，手动选择需要保留的冲突内容，解决冲突后，保存文件，最后提交即可

        # 提交解决冲突后的代码
        git add .
        git commit -m "解决冲突后的提交说明"
        # 查看状态，确认解决冲突后的代码已提交
        git status
        # nothing to commit, working tree clean

rebase 需要用 --continue 继续操作

        git add 文件名
        # 继续 rebase 操作
        git rebase --continue

如果还有其他冲突文件，重复以上操作，直到所有冲突文件都解决，如果想放弃本次 rebase 操作，可执行 git rebase --abort

        # 最后验证
        git status
        # nothing to commit, working tree clean

因此开发的时候推荐拆分成多个小提交，而不是一次性提交大量代码，冲突时更容易定位和解决

5. 常用分支名

 核心主分支（仓库的基础分支，长期存在）

不在主分支上直接改动，要保持主分支的稳定性，保证功能完整可用、可部署

  • 生产环境代码 main / master
  • 开发完成的功能代码 develop / dev

main 分支设置为保护分支，不允许直接推送代码，只能通过 Pull Request/Merge Request 合并

一般基于 dev 分支创建新分支，进行开发或修复 bug、发布新功能等

 临时开发分支（完成特定任务后删除，短期存在）

  • 功能分支 feature/xxx 或 feat/xxx，开发新需求、新功能、新特性（基于 dev 分支创建，完成后合并回 dev）
  • 修复分支 bugfix/xxx 或 fix/xxx，基于 dev 分支创建，完成后合并回 dev
  • 线上紧急修复分支 hotfix/xxx 或 hf/xxx，基于 main 分支创建，完成后同时合并回 main 和 dev 分支
  • 发布分支 release/xxx 或 rel/xxx，基于 dev 分支创建，完成后合并回 main 和 dev

例如开发新功能 xxx 时

        # 切换分支到 dev
        git switch dev
        git pull origin dev
        # 基于当前分支创建新分支 feature/xxx
        git branch feature/xxx
        # 切换到新分支
        git switch feature/xxx
        # 开发功能并多次提交（示例）
        git add .
        git commit -m "feat: 完成xxx功能的核心逻辑"
        git add .
        git commit -m "feat: 补充xxx功能的边界条件处理"
        # 合并到 dev
        git switch dev
        git pull origin dev
        git merge feature/xxx --no-ff -m "feat: 合并 feature/xxx：新增 xxx 功能"
        # 推送合并后的 dev 到远程
        git push origin dev
        # 删除 feature/xxx 分支
        git branch -d feature/xxx
        git push origin --delete feature/xxx

流程：基于 dev → feature/xxx + bugfix/yyy → dev，后续需要发版时，基于 dev 创建 release/a.b.c 分支

例如开发完成，达到上线标准，基于 dev 分支创建上线分支 release/v1.0.1，推送远程后，通过 Pull Request/Merge Request 合并到 main 分支

        # 拉取 dev 分支最新代码
        git switch dev
        git pull origin dev
        # 基于 dev 分支创建上线分支 release/v1.0.1
        git switch -c release/v1.0.1
        # 推送到远程
        git push origin release/v1.0.1
        # 打开 PR/MR，代码 review 后合并到 main
        # 本地拉取 main 最新代码
        git switch main
        git pull origin main
        # 打 Tag v1.0.1
        git tag -a v1.0.1 -m "v1.0.1 发布：修复 yyy 问题，新增 xxx 功能"
        # 推送 Tag 到远程
        git push origin v1.0.1
        # 切换分支到 dev
        git switch dev
        # 拉取 dev 分支最新代码
        git pull origin dev
        # 合并 main 分支到 dev
        git merge main -m "同步 main 分支上线代码：v1.0.1 修复 yyy+新增 xxx"
        # 若有冲突，手动解决后提交
        # git add . && git commit -m "合并 main 到 dev，解决xxx文件冲突"
        # 推送合并后的 dev 到远程
        git push origin dev
        # 删除本地 release/v1.0.1 分支
        git branch -d release/v1.0.1
        # 删除远程 release/v1.0.1 分支
        git push origin --delete release/v1.0.1

流程：基于 dev → release → main → 同步回 dev

例如在开发时需要紧急修复线上问题 zzz

        # 保存当前修改（在 feature/xxx 分支上）
        git stash save -a "开发 feature/xxx 时临时保存，紧急修复 bugfix/yyy"
        # 从当前分支 feature/xxx 切换分支到 main
        git switch main
        git pull origin main
        # 基于当前分支创建并切换到新分支 hotfix/zzz
        git switch -c hotfix/zzz
        # 修复 bug 并提交
        git add .
        git commit -m "fix: 修复 zzz 问题，解决 xxx 场景的异常情况"
        # 推送 hotfix/zzz 到远程
        git push origin hotfix/zzz
        # 打开 PR/MR，代码 review 后合并到 main
        # 本地拉取 main 最新代码
        git switch main
        git pull origin main
        # 打 Tag v1.0.2（最后一位修复版本号 + 1）
        git tag -a v1.0.2 -m "v1.0.2 发布：修复 zzz 问题"
        git push origin v1.0.2
        # 切换分支到 dev
        git switch dev
        # 拉取 dev 分支最新代码
        git pull origin dev
        # 合并 main 分支到 dev（也可以直接合并 hotfix/zzz 分支）→ 需要写明合并来源
        git merge main -m "同步 main 分支的 hotfix：v1.0.2 修复线上 zzz 问题"
        # 推送合并后的 dev 到远程
        git push origin dev
        # 删除 hotfix/zzz 分支
        git branch -d hotfix/zzz
        git push origin --delete hotfix/zzz
        # 切换回 feature/xxx 分支
        git switch feature/xxx
        # 恢复之前保存的代码
        # 可以使用 git stash list 查看 stash 列表
        git stash pop stash编号
        # 切换回 dev 分支，假设合并时有冲突需要解决
        git switch dev
        git pull origin dev
        # 合并 feature/xxx 分支到 dev 分支
        git merge feature/xxx -m "合并 feature/xxx 分支到 dev 分支，新增 xxx 功能"
        # CONFLICT: 合并时发生冲突，需要手动解决，解决后提交
        git add .
        git commit -m "合并 feature/xxx 分支到 dev 分支，解决冲突"
        # 推送合并后的 dev 到远程
        git push origin dev
        # 删除 feature/xxx 分支
        git branch -d feature/xxx
        git push origin --delete feature/xxx

流程：基于 main → hotfix/zzz → main → 同步回 dev

注意：

  1. 切换到主分支/线上存在的分支，需要先拉取最新代码
  2. 合并时可能会发生冲突，需要手动解决
  3. 合并完成后，及时删除临时分支

6. Conventional Commits（约定式提交） 规范

 feat - 新增功能（feature）

  • feat: 新增用户登录功能
  • feat: 实现商品列表分页查询
  • feat: 添加微信支付回调接口

 fix - 修复 Bug

  • fix: 修复登录验证码不显示的问题
  • fix: 解决订单金额计算错误 Bug
  • fix: 修复线上支付失败的紧急问题

 docs - 文档更新

  • docs: 更新 README 中的部署步骤
  • docs: 补充接口文档的参数说明
  • docs: 完善代码中的注释说明

 style - 代码格式调整（不影响功能）

  • style: 格式化代码缩进（符合 ESLint 规范）
  • style: 调整变量命名，统一驼峰式
  • style: 删除多余的空行和注释

 refactor - 代码重构（不影响功能）

  • refactor: 重构用户服务模块，提取公共工具类
  • refactor: 优化接口请求封装，简化代码逻辑
  • refactor: 拆分超大组件，提高可维护性

 test - 测试相关

  • test: 为用户登录功能添加单元测试
  • test: 完善订单模块的集成测试用例
  • test: 修复测试代码中的断言错误

 chore - 构建 / 工程配置 / 辅助工具变更（不涉及业务代码）

  • chore: 升级 axios 依赖到 1.6.0 版本
  • chore: 配置 Webpack 打包优化参数
  • chore: 添加 .gitignore 忽略日志文件
  • chore: 更新 ESLint 配置规则

 perf - 性能优化

  • perf: 优化商品列表查询速度，添加缓存
  • perf: 减少 DOM 操作，提升页面渲染性能

 revert - 回滚提交

  • revert: 回滚 "feat: 新增用户登录功能"
  • revert: 撤销 fix: 修复订单金额计算错误 Bug

 ci - 持续集成 / 持续部署配置变更

  • ci: 配置 GitHub Actions 自动打包部署
  • ci: 调整 GitLab CI 测试步骤的超时时间

7. GitHub Workflow

GitHub Workflow 是 GitHub Actions 的核心组成部分，通过仓库中 .github/workflows/ 目录下的 YAML 文件定义自动化流程，可实现代码检查、打包、部署、发布等自动化任务。

 工作流文件

工作流文件为 YAML 格式（后缀为 .yml 或 .yaml），存放于仓库的 .github/workflows 目录下，文件名可自定义（如 ci.yml、deploy.yml、release.yml 等），一个仓库可包含多个工作流文件。

 工作流核心组成

一个完整的工作流文件通常包含以下核心字段：

  • name：工作流的名称，会显示在 GitHub 仓库的 Actions 面板中，可选字段
  • on：触发工作流的事件（必填），支持 push（代码推送）、pull_request（PR 提交）、schedule（定时任务）、release（发布事件）或 tags（标签推送）等
  • permissions：定义工作流运行时的权限（如 contents: write 允许写入仓库内容），可选字段
  • jobs：工作流的核心任务集合，每个 job 代表一个独立的执行单元，默认并行运行
  • jobs..runs-on：指定 job 运行的环境（如 ubuntu-latest、windows-latest），必填
  • jobs..needs：指定 job 的依赖关系，控制任务串行执行（如依赖其他 job 完成后再运行）
  • jobs..steps：每个 job 包含的具体操作步骤（如运行命令、调用 Action、上传产物等）
  • jobs..environment：关联 GitHub 环境配置（如部署前审批、环境专属变量），非运行环境
  • secrets：仓库/组织级别的敏感信息（如 API 密钥、Token），需在 steps 中引用（非工作流顶层字段）

 工作流触发事件

工作流通过 on 字段触发，常见触发场景：

  • 代码推送/PR：push: { branches: ["main"] }、pull_request: { types: [opened, synchronize] }
  • 标签推送：push: { tags: ["v*"] }（匹配 v1.0、v2.1 等版本标签）
  • 定时任务：schedule: { cron: "0 0 * * *" }（UTC 时间每天凌晨执行）
  • 手动触发：workflow_dispatch:（在 Actions 面板手动启动）

 工作流任务（Jobs）

一个工作流可定义多个 jobs，默认并行执行；通过 needs 字段可指定 job 依赖，实现串行执行。每个 job 包含多个 steps，步骤支持：

  • 运行 shell 命令（run 字段）；
  • 调用官方/第三方 Action（uses 字段）；
  • 上传/下载构建产物（Artifacts）；
  • 设置环境变量、秘钥等。

 示例：标签触发的打包发布工作流

        # 工作流名称：标签发布
        name: tag release
        
        # 触发条件：推送以 v 开头的标签（如 v1.0.0）
        on:
          push:
            tags:
              - "v*"
        
        # 权限配置：允许写入仓库内容（用于创建 Release）
        permissions:
          contents: write
        
        jobs:
          # 第一个任务：打包项目文件
          package:
            runs-on: ubuntu-latest # 运行在 Ubuntu 最新版 Runner 上
            steps:
              - name: Checkout code
                uses: actions/checkout@v4 # 拉取仓库代码
        
              - name: Package files
                run: |
                  # 创建临时目录，避免竞态条件
                  mkdir -p /tmp/ethereal
                  cp -r . /tmp/ethereal/
                  # 清理不需要打包的目录
                  rm -rf /tmp/ethereal/.github /tmp/ethereal/dist /tmp/ethereal/.git
                  # 打包为 tar.gz 压缩包
                  tar -czvf ethereal.tar.gz -C /tmp/ethereal .
        
              - name: Upload artifact
                uses: actions/upload-artifact@v4 # 上传打包文件为产物
                with:
                  name: ethereal # 产物名称
                  path: ethereal.tar.gz # 产物路径
        
          # 第二个任务：创建 GitHub Release
          release:
            runs-on: ubuntu-latest
            needs: package # 依赖 package 任务完成后执行
            steps:
              - name: Download artifacts
                uses: actions/download-artifact@v4 # 下载打包产物
                with:
                  name: ethereal
                  path: . # 下载到当前目录
        
              - name: Create GitHub release
                uses: softprops/action-gh-release@v2 # 调用 Release 生成 Action
                with:
                  generate_release_notes: true # 自动生成发布说明
                  files: |
                    ethereal.tar.gz # 附加打包文件到 Release
