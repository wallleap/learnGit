Git 学习

1、初始化环境

 下载 Git、注册 GitHub 等账号

          # 查看 git 版本
          git --version

 配置本地 Git 信息

  • git config --global user.name "Your Name"
  • git config --global user.email "email@example.com"
  •               # 查看配置信息（按 q 退出）
              git config --list

添加了 --global 是针对全局生效的

如果某个仓库需要单独的，只需在这个仓库中使用没有 --global 的命令配置一遍即可，或把 --global 换成 --local

 生成 keygen

ssh-keygen -t rsa -C "email@example.com"

或使用 ed25519 算法

ssh-keygen -t ed25519 -C "SSH Key"

将生成的 ~/.ssh/id_rsa.pub / ~/.ssh/id_ed25519.pub 中的公钥放到需要提交的服务器上（追加到 ~/.ssh/authorized_keys 文件中）或者在线平台的设置 SSH keys 新建一个，拷贝公钥内容上去

 测试连接（例如 GitHub）

ssh -T git@github.com

输出如下内容就代表成功

Hi username! You've successfully authenticated, but GitHub does not provide shell access.

2、了解 Git 基本命令

工作区（working tree），暂存区（index /stage），本地仓库（repository），远程仓库（remote）

 本地仓库

初始化本地仓库

git init

创建 .gitignore 文件

          touch .gitignore
          # 修改这个文件，将需要忽略的文件名/目录写入
          vi .gitignore

查看仓库中文件状态

          git status

 暂存区

添加当前目录下文件到暂存区

          git add .
          # 添加部分文件
          git add 文件名|文件夹
          # 从暂存区删除文件，但保留工作区文件
          git rm --cached 文件名|文件夹
          # 恢复指定文件 restore 或用 checkout
          git restore 文件名|文件夹
          git restore .

将暂存区中的文件进行提交

          # 直接在 -m 后写提交说明
          git commit -m "本次提交的说明"
          # 使用默认编辑器编写提交说明，保存后关闭文件即可
          git commit
          # 修正最近一次提交（会保留原提交内容，仅补充文件/修改提交信息）
          git commit --amend

 不提交 保留修改（到存储库）

          git stash -a
          # 查看 stash 列表
          git stash list
          # 恢复并删除 stash
          git stash pop stash编号

 远程仓库

添加远程仓库

          git remote add origin(名字) 仓库链接(ssh或https)

查看远程仓库

          git remote -v

删除远程仓库链接

          git remote remove origin(名字)

推送本地代码到远程仓库

          git push origin main:main
          # origin 是远程仓库别名、前一个 main 是本地分支名、后一个 main 是远程分支名，相同时可省略后半部分
          # 第一次可以多加一个参数 -u，将设置默认推送目标，后续可以直接使用 git push
          git push -u origin main
          # 还有其他参数，-f 可以强制推送，--all 推送所有分支，--tag 推送所有标签

拉取远程代码到工作区

          git pull origin main:main
          # origin 是远程仓库别名、前一个 main 是远程分支名、后一个 main 是本地分支名
          # --rebase 参数使用变基，-r 禁用变基强制使用合并
          # fetch 下载最新更改到本地仓库，但不自动合并到当前分支
          git fetch origin

克隆远程代码到本地仓库

          git clone 仓库链接 目录名
          # -b main 指定克隆分支，--depth 1 指定克隆深度

 查看文件修改内容

          git diff 文件名

 查看历史记录

          git log
          git log --pretty=oneline

 查看引用日志

          git reflog

 版本回退

          # 回退到上一个版本(Git 中 HEAD 代表当前版本)
          git reset --hard HEAD^
          # 回退到上上个版本
          git reset --hard HEAD^^
          # 回退到上100个版本
          git reset --hard HEAD~100
          # 根据版本号回退
          git reset --hard 版本号前几位或写全
          # 撤销指定版本
          git revert 版本号前几位或写全

如果需要“回到”某个版本，应该使用如下操作（从指定版本恢复文件到工作区）

          git restore --source 版本号前几位或写全 .

 标签

          # 列出所有 tag
          git tag
          # 在当前版本新建 tag
          git tag 标签名
          # 带注释的标签
          git tag -a 标签名 -m "注释"
          # 在某个版本新建 tag
          git tag 标签名 版本号
          # 推送 tag 到远程仓库
          git push origin 标签名
          # 推送所有 tag
          git push origin --tags
          # 删除 tag
          git tag -d 标签名
          # 删除远程 tag
          git push origin --delete 标签名
          git push origin :标签名 # 简写形式

 分支管理

          # 创建分支
          git branch 分支名
          # 切换分支
          git checkout 分支名
          git switch 分支名
          # 可以简写为一条
          git checkout -b 分支名
          git switch -c 分支名
          # 切换到上一分支
          git switch -
          # 查看分支，-r 参数查看远程分支，-a 参数查看所有分支，-v 参数查看分支+最后一次提交信息
          git branch
          # 合并分支到当前分支
          git merge 分支名
          # 挑选某个提交合并到当前分支
          git cherry-pick 版本号
          # 推送分支到远程仓库
          git push origin 分支名
          # 删除分支，-d 参数删除已合并分支，-D 参数强制删除本地分支
          git branch -d 分支名
          # 删除远程分支
          git push origin --delete 分支名
          git push origin :分支名 # 简写形式

3. 冲突解决

当在不同分支上对同一个文件同一行（或相邻行）代码进行了不同的修改，合并/变基时（git merge / git rebase）就会发生冲突（CONFLICT），这时候需要手动解决冲突。 

          # 查看冲突文件
          git status
          # Unmerged paths:
          #   (use "git add/rm ..." as appropriate to mark resolution)
          #   both modified:  文件名

打开冲突文件，会看到类似如下内容

          # 冲突部分
          <<<<<<< HEAD
          当前分支的代码
          =======
          待合并分支的代码
          >>>>>>> 分支名

解决冲突的方法有很多种，可以将冲突部分的内容修改为自己需要的内容，然后删除 <<<<<<< HEAD、 =======、 >>>>>>> 分支名 这三行，最后提交。

或者使用编辑器（如 VS Code）打开文件，手动选择需要保留的冲突内容，解决冲突后，保存文件，最后提交即可

          # 提交解决冲突后的代码
          git add .
          git commit -m "解决冲突后的提交说明"
          # 查看状态，确认解决冲突后的代码已提交
          git status
          # nothing to commit, working tree clean

rebase 需要用 --continue 继续操作

          git add 文件名
          # 继续 rebase 操作
          git rebase --continue

如果还有其他冲突文件，重复以上操作，直到所有冲突文件都解决，如果想放弃本次 rebase 操作，可执行 git rebase --abort

          # 最后验证
          git status
          # nothing to commit, working tree clean

因此开发的时候推荐拆分成多个小提交，而不是一次性提交大量代码，冲突时更容易定位和解决

5. 常用分支名

 核心主分支（仓库的基础分支，长期存在）

不在主分支上直接改动，要保持主分支的稳定性，保证功能完整可用、可部署

  • 生产环境代码 main / master
  • 开发完成的功能代码 develop / dev

main 分支设置为保护分支，不允许直接推送代码，只能通过 Pull Request/Merge Request 合并

一般基于 dev 分支创建新分支，进行开发或修复 bug、发布新功能等

 临时开发分支（完成特定任务后删除，短期存在）

  • 功能分支 feature/xxx 或 feat/xxx，开发新需求、新功能、新特性（基于 dev 分支创建，完成后合并回 dev）
  • 修复分支 bugfix/xxx 或 fix/xxx，基于 dev 分支创建，完成后合并回 dev
  • 线上紧急修复分支 hotfix/xxx 或 hf/xxx，基于 main 分支创建，完成后同时合并回 main 和 dev 分支
  • 发布分支 release/xxx 或 rel/xxx，基于 dev 分支创建，完成后合并回 main 和 dev

例如开发新功能 xxx 时

          # 切换分支到 dev
          git switch dev
          git pull origin dev
          # 基于当前分支创建新分支 feature/xxx
          git branch feature/xxx
          # 切换到新分支
          git switch feature/xxx
          # 开发功能并多次提交（示例）
          git add .
          git commit -m "feat: 完成xxx功能的核心逻辑"
          git add .
          git commit -m "feat: 补充xxx功能的边界条件处理"
          # 合并到 dev
          git switch dev
          git pull origin dev
          git merge feature/xxx --no-ff -m "feat: 合并 feature/xxx：新增 xxx 功能"
          # 推送合并后的 dev 到远程
          git push origin dev
          # 删除 feature/xxx 分支
          git branch -d feature/xxx
          git push origin --delete feature/xxx

流程：基于 dev → feature/xxx + bugfix/yyy → dev，后续需要发版时，基于 dev 创建 release/a.b.c 分支

例如开发完成，达到上线标准，基于 dev 分支创建上线分支 release/v1.0.1，推送远程后，通过 Pull Request/Merge Request 合并到 main 分支

          # 拉取 dev 分支最新代码
          git switch dev
          git pull origin dev
          # 基于 dev 分支创建上线分支 release/v1.0.1
          git switch -c release/v1.0.1
          # 推送到远程
          git push origin release/v1.0.1
          # 打开 PR/MR，代码 review 后合并到 main
          # 本地拉取 main 最新代码
          git switch main
          git pull origin main
          # 打 Tag v1.0.1
          git tag -a v1.0.1 -m "v1.0.1 发布：修复 yyy 问题，新增 xxx 功能"
          # 推送 Tag 到远程
          git push origin v1.0.1
          # 切换分支到 dev
          git switch dev
          # 拉取 dev 分支最新代码
          git pull origin dev
          # 合并 main 分支到 dev
          git merge main -m "同步 main 分支上线代码：v1.0.1 修复 yyy+新增 xxx"
          # 若有冲突，手动解决后提交
          # git add . && git commit -m "合并 main 到 dev，解决xxx文件冲突"
          # 推送合并后的 dev 到远程
          git push origin dev
          # 删除本地 release/v1.0.1 分支
          git branch -d release/v1.0.1
          # 删除远程 release/v1.0.1 分支
          git push origin --delete release/v1.0.1

流程：基于 dev → release → main → 同步回 dev

例如在开发时需要紧急修复线上问题 zzz

          # 保存当前修改（在 feature/xxx 分支上）
          git stash save -a "开发 feature/xxx 时临时保存，紧急修复 bugfix/yyy"
          # 从当前分支 feature/xxx 切换分支到 main
          git switch main
          git pull origin main
          # 基于当前分支创建并切换到新分支 hotfix/zzz
          git switch -c hotfix/zzz
          # 修复 bug 并提交
          git add .
          git commit -m "fix: 修复 zzz 问题，解决 xxx 场景的异常情况"
          # 推送 hotfix/zzz 到远程
          git push origin hotfix/zzz
          # 打开 PR/MR，代码 review 后合并到 main
          # 本地拉取 main 最新代码
          git switch main
          git pull origin main
          # 打 Tag v1.0.2（最后一位修复版本号 + 1）
          git tag -a v1.0.2 -m "v1.0.2 发布：修复 zzz 问题"
          git push origin v1.0.2
          # 切换分支到 dev
          git switch dev
          # 拉取 dev 分支最新代码
          git pull origin dev
          # 合并 main 分支到 dev（也可以直接合并 hotfix/zzz 分支）→ 需要写明合并来源
          git merge main -m "同步 main 分支的 hotfix：v1.0.2 修复线上 zzz 问题"
          # 推送合并后的 dev 到远程
          git push origin dev
          # 删除 hotfix/zzz 分支
          git branch -d hotfix/zzz
          git push origin --delete hotfix/zzz
          # 切换回 feature/xxx 分支
          git switch feature/xxx
          # 恢复之前保存的代码
          # 可以使用 git stash list 查看 stash 列表
          git stash pop stash编号
          # 切换回 dev 分支，假设合并时有冲突需要解决
          git switch dev
          git pull origin dev
          # 合并 feature/xxx 分支到 dev 分支
          git merge feature/xxx -m "合并 feature/xxx 分支到 dev 分支，新增 xxx 功能"
          # CONFLICT: 合并时发生冲突，需要手动解决，解决后提交
          git add .
          git commit -m "合并 feature/xxx 分支到 dev 分支，解决冲突"
          # 推送合并后的 dev 到远程
          git push origin dev
          # 删除 feature/xxx 分支
          git branch -d feature/xxx
          git push origin --delete feature/xxx

流程：基于 main → hotfix/zzz → main → 同步回 dev

注意：

  1. 切换到主分支/线上存在的分支，需要先拉取最新代码
  2. 合并时可能会发生冲突，需要手动解决
  3. 合并完成后，及时删除临时分支

6. Conventional Commits（约定式提交） 规范

 feat - 新增功能（feature）

  • feat: 新增用户登录功能
  • feat: 实现商品列表分页查询
  • feat: 添加微信支付回调接口

 fix - 修复 Bug

  • fix: 修复登录验证码不显示的问题
  • fix: 解决订单金额计算错误 Bug
  • fix: 修复线上支付失败的紧急问题

 docs - 文档更新

  • docs: 更新 README 中的部署步骤
  • docs: 补充接口文档的参数说明
  • docs: 完善代码中的注释说明

 style - 代码格式调整（不影响功能）

  • style: 格式化代码缩进（符合 ESLint 规范）
  • style: 调整变量命名，统一驼峰式
  • style: 删除多余的空行和注释

 refactor - 代码重构（不影响功能）

  • refactor: 重构用户服务模块，提取公共工具类
  • refactor: 优化接口请求封装，简化代码逻辑
  • refactor: 拆分超大组件，提高可维护性

 test - 测试相关

  • test: 为用户登录功能添加单元测试
  • test: 完善订单模块的集成测试用例
  • test: 修复测试代码中的断言错误

 chore - 构建 / 工程配置 / 辅助工具变更（不涉及业务代码）

  • chore: 升级 axios 依赖到 1.6.0 版本
  • chore: 配置 Webpack 打包优化参数
  • chore: 添加 .gitignore 忽略日志文件
  • chore: 更新 ESLint 配置规则

 perf - 性能优化

  • perf: 优化商品列表查询速度，添加缓存
  • perf: 减少 DOM 操作，提升页面渲染性能

 revert - 回滚提交

  • revert: 回滚 "feat: 新增用户登录功能"
  • revert: 撤销 fix: 修复订单金额计算错误 Bug

 ci - 持续集成 / 持续部署配置变更

  • ci: 配置 GitHub Actions 自动打包部署
  • ci: 调整 GitLab CI 测试步骤的超时时间
